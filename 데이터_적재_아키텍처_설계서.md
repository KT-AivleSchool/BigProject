# [아키텍처 설계서] 데이터 적재(ETL) 업로드 방식 및 UI/UX 설계안

본 설계서는 공무원(사용자)이 외부 공공 데이터를 다운로드받아 시스템에 주입할 때, **"모든 데이터를 한 번에 ZIP 일괄 업로드하는 방식"**과 **"데이터 항목별로 개별 업로드하는 방식"**의 기술적 트레이드오프를 평가하여 스마트시티 SDSS 플랫폼에 가장 적합한 아키텍처를 결정하고 설계한 문서입니다.

---

## 1. 두 업로드 방식의 비교 분석 및 결정

| 평가 항목 | 일괄 업로드 방식 (Monolithic Bulk ZIP) | 항목별 개별 업로드 방식 (Granular Itemized) |
| :---: | :--- | :--- |
| **시스템 적합성** | 🔴 **부적합** | 🟢 **최적합 (최종 채택)** |
| **데이터 변경 주기** | 🔴 **행정 낭비** (10개 데이터를 매번 묶어 올림) | 🟢 **실무 최적화** (조례나 생활인구 등 필요한 파일만 독립 갱신) |
| **개발 구현 난이도** | 🔴 **지옥** (압축 해제, 파일명 검증, 예외 복구 복잡) | 🟢 **쉬움** (각 파일마다 1:1 전용 검증 API 엔드포인트 제공) |
| **대용량 파일 안정성** | 🔴 **위험** (수백 MB의 ZIP 전송 시 메모리 다운/타임아웃) | 🟢 **안전** (개별 파일 독립 전송으로 네트워크 부하 최소화) |
| **사용자 피드백 (UX)** | 🔴 **불명확** (압축 내부 에러 발생 시 원인 추적 불가) | 🟢 **명확** (특정 파일 포맷 에러 시 즉시 경고창 및 가이드 제시) |

### 💡 최종 아키텍처 결정 사유
1.  **데이터 업데이트 주기의 비대칭성:**
    *   `연속지적도`, `학교/어린이집 위치`는 1년에 1번 업데이트되는 정적 데이터입니다.
    *   `생활인구`, `교통 통계`, `민원 데이터`는 월/주 단위로 자주 갱신되는 유동적 데이터입니다.
    *   변경 주기가 완전히 다른데 매번 10개 대용량 데이터를 하나의 ZIP 파일로 통째로 묶어 올리게 강제하는 것은 실무 행정상 엄청난 낭비이자 오작동의 근원입니다.
2.  **주니어 개발 생산성 보장 (7주 일정 준수):**
    *   일괄 ZIP 적재 방식은 압축 해제 시 OS별 인코딩 깨짐(한글 깨짐), 파일명 유효성 체크, 일부 파일 누락 시 롤백(Rollback) 트랜잭션 처리 등 구현 난이도가 너무 높습니다.
    *   **항목별 개별 업로드 방식**은 `/api/v1/upload/nosmoking` 같이 API를 쪼갤 수 있어, 주니어 D(API)가 단일 파일 규격만 검증하는 단순 코드를 작성하면 되므로 1주일 이내로 개발이 완료됩니다.

### ⚙️ 백엔드(FastAPI) 업로드 연동 프로세스
1.  **업로드 ➔ 검증:** 공무원이 `6번 행정동 공간경계` 파일 업로드 시, 백엔드는 해당 파일의 헤더 칼럼에 `동코드`, `geometry` 공간 객체가 정상적으로 들어있는지 1차 유효성 체크를 가동합니다.
2.  **공간 보정:** 이상이 없다면 GeoPandas가 즉시 CRS 좌표계를 `EPSG:4326` 위경도로 일관성 있게 투영 변환합니다.
3.  **데이터 스왑(Swap):** PostgreSQL DB 상에서 `TRUNCATE TABLE seoul_dong_boundary;`를 가동하여 기존 데이터를 깔끔하게 날린 뒤, 신규 업로드된 파일 데이터를 고속 Insert(`to_sql`)하고 공간 인덱스(`GIST`)를 다시 걸어 즉시 사용 가능한 상태로 스위칭합니다.
