# [데이터 가공 명세서] 공공데이터 정제 및 PostGIS 적재 가이드라인

본 가이드라인은 주니어 C(데이터)와 주니어 D(API)가 수집한 **10대 데이터셋의 비정형/미정제 자료를 PostgreSQL(PostGIS) 공간 DB에 저장 및 GIS 연산이 가능한 데이터 모델로 가공하는 구체적인 정제 기법과 형태**를 기술한 개발 표준 가이드북입니다.

---

## 1. 데이터 가공 대상 및 변환 형태 정의

우리가 가공해야 할 10대 데이터셋 중, GIS 연산 및 시뮬레이션을 위해 반드시 전처리가 필요한 핵심 가공 대상은 다음 4가지 그룹으로 분류됩니다.

```
[원천 공공데이터]                       [전처리/가공 작업]                  [가공 완료 형태 (DB)]
1. 텍스트 주소록     ───────────────>  Geocoding API 변환  ───────────────>  Point Geometry (4326)
2. 5186/5174 SHP 지도  ───────────────>  GeoPandas to_crs()  ─────────────>  WGS84 Polygon (4326)
3. 교통량 통계 + 좌표   ───────────────>  Pandas Inner Join   ─────────────>  유동인구 공간 노드
4. 시계열 생활인구      ───────────────>  시간대/요일 Aggregation  ─────────>  경량화 요약 인구 테이블
```

---

## 2. 세부 데이터셋별 가공 프로세스 가이드

### 1️⃣ 지리적 텍스트 주소 ➔ 공간 지오메트리(Point Geometry) 변환
*   **가공 대상 데이터:** `1번 금연구역 정보`, `2번 어린이집 정보` 중 일부 위경도 누락 데이터 및 `7번 상가정보`
*   **가공이 필요한 이유:** 공공데이터 중 도로명 주소만 적혀 있고 위경도가 `null`이거나 오기된 행이 있어 그대로 적재 시 지도 상에 렌더링되지 않고 공간 연산(Buffer)에서 유실됩니다.
*   **가공 방법 (Geocoding):**
    1.  위경도가 비어 있는 행의 주소(Address) 컬럼을 추출합니다.
    2.  카카오/네이버 지도 Geocoding API 또는 공공 주소전환 API를 호출하여 위경도 좌표 `(x, y)`를 획득합니다.
    3.  획득된 위경도를 파이썬 `shapely.geometry.Point` 객체로 감싸고, 공간 좌표계를 `EPSG:4326`으로 매핑하여 DB의 `geometry(Point, 4326)` 타입 컬럼에 인서트합니다.

### 2️⃣ 공간 좌표계 투영 통일 (Coordinate Projection Standardization)
*   **가공 대상 데이터:** `6번 서울시 행정동 경계 SHP`, `9번 국토부 연속지적도 SHP`
*   **가공이 필요한 이유:** 국토부 지적도는 보통 중부원점 좌표계(`EPSG:5186` 등)를 쓰고, 버스정류장/어린이집은 WGS84 위경도(`EPSG:4326`)를 씁니다. 좌표계를 통일하지 않으면 PostGIS에서 공간 결합 연산 수행 시 모든 지점이 엉뚱한 바다 한가운데에 찍히게 됩니다.
*   **가공 방법 (CRS Alignment):**
    *   GeoPandas 라이브러리를 이용하여 지적도 SHP 파일을 로드한 후, `.to_crs(epsg=4326)` 명령어를 사용하여 공간 투영계를 일괄 위경도로 변환 가공하여 파일로 저장하거나 DB에 적재합니다.

### 3️⃣ 대중교통 노드 좌표 및 월평균 교통량 조인 (Data Normalization)
*   **가공 대상 데이터:** `3번 교통 노드 좌표 마스터` + `4번 교통 이용객 통계 정보`
*   **가공이 필요한 이유:** 이용객 수치 통계 데이터에는 위경도가 없고, 좌표 마스터 데이터에는 승객 수 수치가 없습니다. 두 테이블을 하나로 합쳐야 **"이용객이 많은 교통 핫스팟 노드"**를 지도에 뿌리고 가중치 점수를 줄 수 있습니다.
*   **가공 방법 (Inner Join & Aggregation):**
    1.  Pandas를 이용하여 일별 승하차 통계 데이터를 `정류소 번호(BUS_STA_NO)` 및 `역 코드(STATION_CD)` 기준으로 월 평균 또는 총합으로 Group By 연산합니다.
    2.  정류소/지하철 위치 마스터 데이터프레임과 `BUS_STA_NO`를 기준으로 이너 조인(`pd.merge`)을 가동합니다.
    3.  합쳐진 데이터프레임의 X, Y 좌표 필드를 이용하여 `gpd.points_from_xy()`로 공간 객체화하여 적재합니다.

### 4️⃣ 대용량 시계열 생활인구 데이터의 경량화 요약 (Dimensionality Reduction)
*   **가공 대상 데이터:** `5번 행정동단위 서울 생활인구`
*   **가공이 필요한 이유:** 매시간, 매동별로 1년 365일 쌓이는 생활인구 데이터는 수천만 건에 달합니다. 웹 서버에서 이를 그대로 공간 조회하면 성능이 심각하게 느려집니다.
*   **가공 방법 (Temporal Feature Aggregation):**
    1.  수집된 데이터를 **요일구분(주중/주말)**과 **시간대구분(출퇴근 시간: 07~09, 18~20 / 낮 시간: 10~17 / 밤 시간: 21~06)**으로 라벨링(Feature Engineering)합니다.
    2.  `행정동코드(dong_code)`, `요일구분`, `시간대구분`으로 데이터를 그룹화한 뒤, 인구 수의 평균(`mean()`)을 산출하여 1개 행정동당 단 6개의 정적 평균 수치 레코드로 축소 가공합니다. 이 작업을 거치면 DB 탐색 속도가 100배 이상 빨라집니다.
