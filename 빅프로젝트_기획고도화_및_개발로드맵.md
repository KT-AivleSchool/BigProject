## 1. 고도화된 과제 정의서 (Refined Proposal)

### 과제 기본 정보
*   **분야**: 스마트 시티 및 과학적 행정 (B2G - Business to Government)
*   **과제명**: **GIS 공간 빅데이터 및 Multi-Agent 시뮬레이션 기반 스마트 시티 실외 흡연구역 최적 입지 선정 및 정책 검증 플랫폼 (SDSS)**
*   **조원**: 배종현 (외 2반 4조원)

### 과제 선정 배경 및 필요성 (공식 통계 및 학술적 근거)
1.  **공간적 갈등의 임계점 도달 (흡연권 vs 혐연권)**:
    *   **현상**: 실외 금연구역은 지속적으로 확대되는 반면 지정 흡연구역은 부족하여, 골목길/보도 등 '회색지대'에서의 비정형 흡연 및 간접흡연 피해 민원이 급증하고 있음.
    *   **통계 근거**: 국민권익위원회 민원 통계(2024)에 따르면 최근 5년간(2020~2024년) 공동주택 및 공공장소 간접흡연 민원은 누적 **51만 건**에 달하며, 2024년 한 해에만 약 16만 7,000건이 접수되어 2020년 대비 **2.4배 급증**함.
    *   **물리적 확산 근거**: 질병관리청-연세대학교 환경공해연구소 연구(2022)에 따르면 실외 흡연 시 담배 유해물질은 **최소 10m에서 최대 100m까지 확산**되어 대기 질을 유의미하게 훼손하므로 정교한 GIS 공간적 버퍼 필터링이 필요함.
2.  **부서 간 칸막이 행정 및 현장 대응의 한계**:
    *   **현상**: 흡연시설 설치 시 보건소, 도시계획과, 환경과 등 여러 부서 간의 복잡한 조율 절차와 기하급수적으로 폭증하는 민원으로 인해 현장 행정이 병목 현상을 겪고 있음.
    *   **통계 근거**: 국토교통부 공동주택 관리 행정 통계(2024)에 따르면 민원 제기 대비 행정 주체의 실제 사실조사 착수율은 2020년 **98.5%**에서 2024년 **54.5%**로 거의 절반 가까이 폭락하여 데이터 기반 과학행정 자동화(SDSS) 도입이 시급함.
3.  **데이터 부재로 인한 중복 예산 낭비**:
    *   **현상**: 주민 반발(NIMBY) 및 법적 규제를 사전 검토하지 않은 주관적 입지 선정으로, 설치 후 철거하거나 재이전하는 매몰 비용 사례가 빈번함.
    *   **통계 근거**: 정화/밀폐형 실외 흡연부스 설치 단가는 개당 **2,000만~3,000만 원** 수준에 달하며, 잘못된 입지 선정으로 인한 철거/이전 발생 시 공공 재정의 **100% 매몰 손실**이 발생함. 이를 차단하기 위한 사전 시뮬레이션 검증이 강력히 요구됨.

### 2단계 의사결정 파이프라인 (2-Stage Decision Pipeline)
본 시스템은 공공 데이터의 객관적 분석(정량)과 이해관계자의 이해조율(정성)을 순차적으로 결합한 **2단계 하이브리드 의사결정 아키텍처**를 가집니다.
1.  **1단계: 정량적 입지 최적화 (MCDA-AHP)**: 공공 데이터를 기반으로 법적 금연구역을 100% 사전 필터링하고 유동인구, 민원 발생 핫스팟 등의 레이어를 연산하여 정량적 가중치 점수를 도출합니다. (AHP 방법론 준용)
2.  **2단계: 정성적 갈등 시뮬레이션 (Multi-Agent)**: 1단계 후보지 정보를 입력받아, 주민 반발이나 시의회 설득 시 발생 가능한 논쟁을 가상 에이전트(도시계획관, 보건행정관, 시민대표) 간의 시뮬레이션을 통해 사전에 스크리닝합니다.

### 주요 서비스 기능 (Web Service Features)
1.  **다부서 협업형 행정 워크플로우 및 의사결정 이력 관리**: 관련 부서 담당자가 공동으로 입지 선정을 검토하고 승인 프로세스를 진행할 수 있는 부서별 권한 제어 및 행정 의사결정 이력 관리 시스템.
2.  **GIS 기반 다기준 입지 분석(MCDA) 대시보드**: 법적 제한 구역(학교, 어린이집 반경 30m 등) 버퍼 존 시각화 및 유동인구 Heatmap, 민원 핫스팟 등 공간 빅데이터 레이어 중첩(Overlay) 분석 대시보드.
3.  **비정형 민원 텍스트 위치 정제 및 공간 집계(Aggregation) 파이프라인**: 국민신문고 등에서 유입되는 비정형 민원 텍스트에서 자연어 처리(NLP)를 통해 위치 정보를 추출하고, 이를 좌표화하여 공간 데이터 대시보드에 자동 맵핑.
4.  **행정 문서 규격 맞춤형 타당성 보고서(One-Click PDF) 반출**: 1단계의 정량적 분석 점수(XAI 그래프)와 2단계의 Multi-Agent 토론 및 리스크 요약 결과를 공공 보고서 양식에 맞춘 정형화된 PDF 문서로 자동 생성하여 행정 결재용 근거 자료로 활용.

## 2. 기술 크리틱 대응 및 방어 전략 (Technical Defense Strategy)

*   **Q1. LLM 환각(Hallucination)이 있는 결과를 공공 문서의 객관적 근거로 쓸 수 있는가?**
    *   *방어 논리*: 최종 의사결정을 자동 집행하는 주체가 아닌 **'의사결정 보조 시스템(SDSS)'**입니다. 법적 위반 여부는 1단계에서 **QGIS/PostGIS 공간 연산 기반의 Rule-based Filter로 100% 필터링**하므로 법적 위반 가능성이 0%입니다. Multi-Agent 시뮬레이션은 조례 및 법령집을 **RAG DB로 바인딩**하여 엄격히 규제된 프롬프트 범위 내에서만 정성적 토론(NIMBY 예방 스트레스 테스트)을 수행하며, 발언마다 법률 주석(Citation)을 강제하여 환각을 제어합니다.
*   **Q2. 딥러닝/공간 분석 모델과 LLM 간의 XAI 가중치(%) 기여도가 어떻게 매핑되는가? (오버엔지니어링 의혹)**
    *   *방어 논리*: 두 기술의 강제 결합이 아닌 **2단계 순차 파이프라인(2-Stage)** 구조입니다. 1단계에서 AHP 수리 모델로 수치적 기여도(XAI)를 도출해 내고, 2단계에서 해당 정량 수치를 LLM 에이전트의 Context로 주입하여, 각 페르소나 에이전트가 해당 정량 데이터를 기반으로 비판과 합의를 도출하게 됩니다. (LLM이 수학적 계산을 직접 수행하는 것이 아님)
*   **Q3. "우리 집 앞 골목에 담배 연기가 심해요" 같은 비정형 텍스트에서 어떻게 정밀 좌표를 추출하는가?**
    *   *방어 논리*: **NER 위치 개체명 추출 + Fuzzy Matching + 공간 인덱스 집계(Aggregation)**를 조합한 3단계 파이프라인을 사용합니다. **KoBART-NER 모델**로 문맥상의 지명을 1차 추출하고, 이를 상가/건물명 마스터 DB와 **Fuzzy String Matching**으로 보정한 후, Vworld 지오코더 API를 통해 정확한 위경도 좌표로 변환합니다. 변환 후 우버의 **H3 공간 인덱싱 시스템을 사용해 육각형 그리드 단위로 Aggregation(집계)**하여 분석하므로 위치 노이즈를 효과적으로 상쇄합니다.
*   **Q4. LLM 에이전트 토론이 합의점에 이르지 못하고 무한 루프에 빠지면 어떻게 하는가?**
    *   *방어 논리*: **Max Round 제약 조건, 중재자(Moderator) 에이전트, Fallback 메커니즘**을 설계에 적용합니다. 토론은 최대 **3라운드**로 강제 제한되며, **Moderator 에이전트**가 각 라운드마다 찬반 쟁점을 정제하고 합의를 유도합니다. 3라운드 종료 시까지 이견이 좁혀지지 않을 경우, **"합의 실패 사유 요약서"**와 **"에이전트 투표(Voting) 결과 및 찬반 대조표"**를 출력하는 Fallback 메커니즘을 작동시킵니다.

## 3. 클라우드 & 쿠버네티스 아키텍처 및 구축 흐름

### ① 마이크로서비스 컨테이너 구성 (Dockerized MSA Pods)
*   **Frontend Pod (React + Nginx)**: 정적 파일 서빙 및 지도 인터페이스 시각화.
*   **Backend API Pod (FastAPI)**: REST API 서빙 및 AHP 입지 최적화 수리 연산 담당.
*   **Redis Pod**: Celery 메시지 브로커 및 캐시 DB.
*   **Celery Worker Pod**: 장시간 소요되는 2단계 Multi-Agent 시뮬레이션 및 RAG 연산을 비동기 백그라운드 태스크로 처리하여 브라우저 타임아웃 방지.
*   **PostGIS DB Pod (PostgreSQL + PostGIS)**: 행정 구역 버퍼, 유동인구, 민원 정보 공간 DB.
*   **vLLM Inference Pod (Solar / Llama-3 sLLM)**: GPU 전용 노드 풀에 배포되며 PagedAttention을 통한 고성능 LLM 추론 전담.

### ② 단계별 구축 프로세스 (Implementation Steps)
1.  **Dockerizing**: 각 컴포넌트별 Dockerfile을 구성하여 경량화 및 환경 일관성 확보.
2.  **AI Serving Setup**: vLLM 오픈소스 이미지에 오픈소스 sLLM 모델 가중치(Weights)를 공유 스토리지(PVC)로 연결하여 API 서빙.
3.  **K8s Configuration**: CPU 전용 일반 노드 풀과 GPU 전용 노드 풀을 나누고, **Tolerations(용인)**과 **Taints(장벽)** 설정을 통해 LLM 추론 컨테이너만 GPU 노드에 할당하여 가성비 극대화.
4.  **Helm Chart Packaging**: 헬름 차트로 전체 아키텍처를 패키징하여 공공기관 행정망(망분리 온프레미스)에 오프라인 패키지 형태로 원클릭 이식할 수 있는 환경 제공.

## 4. 알고리즘 편향 리스크 및 해결방안

### ① 편향 리스크 (Alleyway Bias) 정의
*   수학적 입지 선정 시 금연구역 필터링 및 민원 최소화(Negative Constraints) 조건만 극대화하면, 알고리즘은 본능적으로 가장 민원이 없는 **도시의 외딴 사각지대(골목 안쪽 구석, 방치된 낙후 공터 등)**를 입지로 선정합니다.
*   이는 흡연자들의 접근성이 떨어져 **부스 이용률을 현저히 낮추고**, 가시성이 없는 밀폐 지역 특성상 **치안 불안 및 청소년 탈선의 공간(우범지대화)**으로 번지는 중대한 부작용이 발생합니다.

### ② 기술적/알고리즘적 해결책 (MCDA 보완)
*   **보행 접근성 감쇄 함수 (Accessibility Decay Function) 탑재**: 실제 흡연 민원 및 인구 집중 수요 핫스팟으로부터 **도보 1~2분(반경 50m~100m) 이내**에만 가산점을 부여하고, 이 범위를 벗어나면 적합도 점수가 지수함수적으로 급감하게 설계하여 실효성 보장.
*   **도시 안전 인프라 데이터 중첩 (Safety Data Overlay)**: 지자체의 **관제 CCTV 버퍼 범위** 및 **가로등 조명 버퍼** 레이어를 입지 가중치 연산에 반영하여, CCTV 감시망이 닿는 열린 공공장소만을 후보지로 유도.
*   **Multi-Agent 정성 보정 피드백 루프**: 1단계에서 걸러지지 못한 후미진 골목 후보지에 대해 2단계의 **[시민대표]** 및 **[도시치안관]** 에이전트가 공청회 토론에서 범죄 리스크 및 불안감을 쟁점화하여 입지 부적합 판정을 내리고 대안지를 재검색하도록 피드백 루프(Human-in-the-Loop) 기동.

## 5. 7주 개발 로드맵 및 MVP 범위 타당성 검증 (MVP Feasibility & Timeline)

### 개발 속도(Velocity) 및 리소스 세이브 전략
*   **AI 페어 프로그래밍 효과**: React UI 마크업, FastAPI 라우터 보일러플레이트 코드, DB 스키마 마이그레이션 등 단순 반복 코딩 시간을 AI가 대신 처리해 주어 **전체 개발 공수의 약 40%가 절감**됩니다.
*   **공간적 범위 축소(Spatial Scoping)**: 7주 내 전국구 서비스 데이터 구축은 불가능하므로, 데이터를 **'서울시 마포구' 1개 자치구 범위로 한정**하여 데이터 획득 병목을 방지하고 모델 완성도를 높입니다.

### 7주 개발 마일스톤 (Weekly Milestone)
1.  **1주차 (설계)**: 아키텍처 및 DB 스키마 설계, API 명세서 확정, Git 환경 세팅
2.  **2주차 (데이터)**: 마포구 GIS 및 민원 데이터 전처리 파이프라인 구축 및 지오코딩 검증
3.  **3주차 (코어 AI)**: AHP 기반 MCDA 알고리즘 및 LangChain 기반 Multi-Agent 토론 로직 구현
4.  **4주차 (웹 개발)**: FastAPI 백엔드 라우터 및 Kakao Map 연동 프론트엔드 GIS 대시보드 구현
5.  **5주차 (시스템 통합)**: 프론트엔드-백엔드-Multi-Agent 비동기 연동 및 예외 처리 구현
6.  **6주차 (테스트 및 평가)**: Hit Rate@10(입지 적합도) 및 F1-Score(NER 모델) 성능 지표 검증 및 버그 픽스
7.  **7주차 (고도화 및 데모)**: PDF 보고서 다운로드 기능 완성 및 최종 발표 자료/데모 영상 제작

### MVP 필수 범위 vs Out of Scope 정의

| 구분 | MVP 필수 범위 (7주 내 완료) | Out of Scope (추후 고도화) |
| :--- | :--- | :--- |
| **대상 지역** | **서울시 1개 자치구** (예: 마포구) | 전국 광역자치단체 확장 |
| **공간 분석** | Rule-based 버퍼 필터링 및 **AHP 수리 점수 계산** | 실시간 보행 흐름 분석 및 도로 네트워크 연동 |
| **Multi-Agent** | 상용 LLM API 활용, **3개 에이전트 3라운드 토론** | sLLM 로컬 파인튜닝 및 온프레미스 전용 인프라 배포 |
| **데이터 연동** | 민원 파일 업로드(CSV) 방식 처리 | 국민신문고 실시간 크롤링 및 API 연동 |
| **보고서 출력** | HTML template 기반의 **원클릭 PDF 다운로드** | 사용자 정의 템플릿 편집 및 한글 파일(.hwp) 변환 |

### 타당성 검증 결론
본 프로젝트는 **공간적 범위를 1개 자치구로 좁히고**, 로컬 sLLM 인프라 구축 대신 **상용 API 활용 프롬프트 체인을 통해 3개 에이전트 시뮬레이션을 구현**하므로 **7주라는 기한 내에 100% 현실적으로 구동 가능한 MVP를 완성할 수 있습니다.**
특히, 5주차 연동 단계에서 발생할 수 있는 LLM API 지연 병목은 **비동기 배치(Queue) 및 Celery/Redis 기반 아키텍처**를 적용하여 브라우저 타임아웃 문제를 완벽히 방어할 수 있습니다.
